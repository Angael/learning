Astro web · lesson 0001

What Astro turns into a website

First principle: a browser does not understand an Astro project. It understands files it can request — mostly HTML, CSS, JavaScript, images, and fonts. Astro's build step turns source files into those browser-ready files.

Primary source

Read later, not first: Astro's docs on project structure and pages. The key documented ideas are: src/ is processed by Astro, public/ is copied as static assets, and files in src/pages/ become routes.

The mental model

source
src/pages/index.astro
→
build
npm run build
→
output
dist/index.html

When this repo builds, Astro reads the page source at src/pages/index.astro, runs the Astro parts, and emits a static file under dist/. That dist/ folder is what a static server or nginx can serve.

One distinction to keep

src/ is for files Astro should understand and transform. public/ is for files Astro should leave alone and copy. If you can say that distinction out loud, Astro routing and deployment become much less mysterious.

Tiny task

Open the repo and compare these two files:

src/pages/index.astro
 dist/index.html

Look for one thing that exists in the source but not exactly the same way in the built output. That difference is evidence that a build step happened.

Retrieval check

Without rereading above: in one or two sentences, explain the path from src/pages/index.astro to the page a browser receives.

Ask the teacher

If any word feels slippery — route, static, build, source, output — ask me. Your job is not to pretend it is clear; your job is to make the model solid.
